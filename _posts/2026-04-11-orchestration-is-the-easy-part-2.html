What we put in them, when we turned them on, what we never let them decide

A builder's journal — Part 2

Part 2 — The design question

Which agents, not how many

CrewAI, LangGraph, AutoGen — these solve orchestration: how to get agents passing information reliably. That is a real problem and they solve it well. What they do not address is the design layer underneath: which agents should exist, what should each one own, and what should each one explicitly refuse to decide.

Three agents activated for the build phase, each a Markdown file in /agents/ with a role, a philosophy, a defined scope, and a list of things it defers:

Mira
Curriculum Director
Lesson design, learning science, tip and quiz systems, AI evaluation criteria per lesson
Kai
Product Designer
Every screen, every tap, every emotional moment in the mobile experience
Noa
Technical Architect
Stack selection, AI pipeline, cost modeling, scalability

Two Python scripts handle coordination. squint.py invokes a single agent, loads its context files, calls the API, and saves output to /outputs/. collaborate.py runs pipelines where each agent's output feeds the next automatically. A new lesson flows like this:

Mira designs → Celeste checks AI feasibility → Kai designs the screen → Sam logs the decision

How memory works across sessions

Context files are the memory. Every agent loads a curated set of project documents before responding. The curriculum, the team brief, the decision log travel between sessions and tools. The agents are stateless. The documents are the state.

The missing agent: Dev

If Mira designs a lesson, Celeste reviews its evaluation criteria, and Kai specifies its screen, why can't a fourth agent named Dev build the Angular component? The answer is tool access. The messages API gives text in, text out. Writing files and running commands requires the tool use API, which takes real implementation work. Claude Code already had it, so we used that instead.

The cost was architectural inconsistency: all active agents in one system, the coding agent operating outside it. The right solution — a Dev agent with tool access, completing the pipeline from design to deployed code — does not exist yet. It is the clearest gap in the architecture.

Activation scheduling

Not all agents run from day one. Elena, the legal agent, activates only before scaling payments across jurisdictions — not at the start, when there are no users to protect. Felix, the strategy agent, waits for Riya's retention data before making pricing decisions. Activating agents before their domain is relevant produces confident-sounding advice with no grounding. The schedule is a product management decision applied to the team design.

The coordination

The watcher

A watcher script monitors /outputs/ and triggers review pipelines automatically when new files appear. Quinn finishes the video mapping, Mira reviews it for curriculum fit without any manual handoff. Kai finishes the screen flow, Celeste reviews the feedback display. When both complete, Sam synthesises a summary. One command, walk away, return to six agents having reviewed each other's work.

Individual agents are useful. Coordinated agents are something qualitatively different.

The limits

What they cannot do

Noa produced theory when asked to write code. Not a failure of the agent — a failure of task assignment. An agent asked outside its scope drifts toward its strengths. Tighter scope, not a smarter prompt, is the fix.

More fundamentally: agents cannot observe humans. No system prompt produces the insight of watching a first-time learner sit with the app for twenty minutes. They cannot make the final call. Felix can frame a pricing tradeoff precisely, but the decision belongs to whoever lives with the consequences.

A team of advisors with no decision-maker is just an expensive echo chamber. — Something Sam would say

And they drift when underconstrained. The quality of agent output is almost entirely a function of prompt specificity — which is the same thing that is true of human teams.

What comes next

The rest of the team

The design accounts for roles the product does not need yet. Each has a system prompt and context files, ready to activate when the product reaches the phase their domain governs:

Celeste
Art Evaluation — Phase 1
Activates when real students submit real work. Audits whether AI feedback is actually correct.
Riya
Analytics — Phase 1
Activates when behavioral data exists: funnels, retention, learning effectiveness.
Jordan
Community — Phase 1
Activates when users need support. Carries qualitative signal that data alone cannot.
Sam
Project Manager — Phase 1
Activates when agents need coordinating. Owns the decision log and dependency map.
Felix
Product Strategy — Phase 2
Activates when retention data exists. Makes the pricing call on evidence, not intuition.
Marco
Marketing — Phase 2
Activates when the product is proven: App Store, community outreach, creator partnerships.
Quinn
Content — Phase 2
Activates when content gaps need original production or creator licensing.
Elena
Legal — Phase 3
Activates before scaling across jurisdictions: GDPR, COPPA, AI disclosure.

The measure of the whole system is whether a complete beginner draws a better sphere after one round of feedback than before. Build the loop first. Prove it on one person. Then build the rest.
